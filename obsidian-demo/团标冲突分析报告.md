## 一、冲突等级说明

| 等级 | 说明 |
|------|------|
| 🔴 高风险 | 标准强制要求（**应 / 不得**），与现状直接冲突，需整改 |

## 二、高风险冲突项（🔴 强制条款）

### C1 — JDK 1.8 版本不符合要求

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§5.1 编码语言要求** |
| **原文** | "宜使用 C++17、**Java 11**、Python 3.8 及以上等主流、安全语言和版本，**不得使用已淘汰或存在安全隐患的语言或版本**" |
| **附录依据** | 开发规范 **附录C 符合性调研表** — 第一行"编码语言 / 开发语言使用情况"检查项：要求"使用主流语言（如 C++17、**Java 11**、Python 3.8 及以上）" |
| **修订意见参考** | 文件C 第 1 条：建议"对'不得使用已淘汰或存在安全隐患的语言或版本'表述适当柔化" |
| **现状** | **JDK 1.8**，Oracle 已于 2019 年停止公开更新 |
| **影响** | 附录C调研表第一项将填"否"，适配测试直接不合格 |
| **建议** | ① 近期：支持修订意见推动"Java 11"柔化表述；② 中期：规划升级至 Java 17 LTS（兼容毕昇/Dragonwell 国产 JDK） |

### C2 — 类命名 Q 前缀与 Java 体系冲突

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§5.3 变量命名要求 g）** |
| **原文** | "g）类名首字母，**公有类以 Q 开头，公有函数以 q 开头**" |
| **修订意见参考** | 文件C 第 3 条：该条"可能更偏向特定开发框架（Qt），建议不要作为通用强制要求" |
| **现状** | Java 标准驼峰命名，无 Q 前缀约定；Spring 注解体系与此完全背离 |
| **影响** | 若强制执行需改造所有 public class/method 命名，工程量极大且无实际意义 |
| **建议** | 积极推动此条从"应"降为"可选"，或增加"适用于 Qt 框架"的适用范围限定 |

### C3 — 源码硬编码中文字符

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§5.4 国际化要求 b）** |
| **原文** | "b）**不得在源码中硬编码中文字符**" |
| **辅助条款** | 开发规范 §5.4 d）："文本资源应集中存放于资源文件（如 .ts、.properties）" |
| **附录依据** | 开发规范 **附录C 符合性调研表** — "国际化支持 / 是否支持 UTF-8/BIG5 编码"检查项："不得硬编码中文字符，应使用 .ts/.qm 等格式管理多语言" |
| **现状** | Java 后端代码中大量中文提示语（如 `throw new BusinessException("用户名不存在")`）、日志中文字符串 |
| **影响** | 附录C调研表国际化项填"否"，适配测试国际化维度扣分 |
| **建议** | 将中文字符串抽取到 `src/main/resources/i18n/messages_zh_CN.properties`，通过 Spring `MessageSource` 管理（等价替代标准中 Qt 工具链的 .ts/.qm 方案） |

### C4 — 前后端通信未使用 TLS

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§8 安全管理要求 e）** |
| **原文** | "e）**前后端通信应使用 TLS**" |
| **附录依据** | 开发规范 **附录C 符合性调研表** — "网络安全 / 通信加密机制"检查项："前后端通信应启用 TLS 加密" |
| **修订意见参考** | 文件C 第 5 条："'前后端通信应使用TLS'对于本机进程间通信场景可能不一定适用，建议区分本机通信和跨网络通信" |
| **现状** | 内网 HTTP 部署，Nginx 未配置全部 HTTPS |
| **影响** | 附录C调研表网络安全项填"否"，安全测试（测试规范 §6.6）也将扣分 |
| **建议** | Nginx 出口配置 HTTPS（可用国密 SM2 证书）；本机 IPC 通信引用修订意见争取豁免 |

### C5 — 配置文件明文存储密码

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§8 安全管理要求 d）** |
| **原文** | "d）**不得明文存储任何凭据，应使用加密存储或密钥管理**" |
| **附录依据** | 开发规范 **附录C 符合性调研表** — "凭据管理 / 密码存储方式"检查项："不允许明文存储，应加密或使用密钥管理系统" |
| **现状** | `application.yml` 中明文存储数据库密码、Redis 密码 |
| **影响** | 附录C调研表凭据管理项填"否"；安全测试（测试规范 §6.6 安全性测试）权限/数据保护验证项将不通过 |
| **建议** | 使用 Jasypt 对 Spring Boot 配置加密；Docker 环境改用 Secret 挂载 |

### C6 — 禁止 sudo/pkexec 提权

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§8 安全管理要求 a）** |
| **原文** | "a）含前端交互逻辑的可执行程序**应以普通用户权限运行，不得使用 sudo/pkexec 提权**" |
| **附录依据** | 开发规范 **附录C 符合性调研表** — "安全运行 / 运行权限控制"检查项："不允许 sudo/pkexec 提权，前后端分离，使用 dbus 通信" |
| **现状** | 部署脚本 / 启动脚本中可能存在 sudo 操作，需自查 |
| **建议** | 排查 shell 脚本中的 sudo 调用，改为以应用专属用户身份运行服务 |

## 三、中风险冲突项（🟡 推荐/间接影响）

### C7 — 日志规范差距

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§7.2 应用日志** a）d）e）f）g） |
| **具体条款** | a）"循环体内不打印 INFO"；d）"**日志文件至少保存 15 天**"；e）"应统一日志级别：DEBUG/INFO/WARN/ERROR/FATAL"；f）"**宜采用 JSON 结构化格式**"；g）"日志默认路径：`/var/log/应用名/`" |
| **修订意见参考** | 文件C 第 4 条："日志文件至少保存 15 天的要求较明确，但部分重要系统可能需要更长保存周期，建议增加'按业务和安全审计要求执行'" |
| **现状** | logback 滚动策略可能未设置 15 天下限；日志为文本格式而非 JSON；批处理循环中可能存在大量 INFO 日志 |
| **建议** | 调整 logback 配置保留天数 ≥15；引入 logstash-logback-encoder 输出 JSON；code review 清理循环体内 INFO |

---

### C8 — 软件包签名和校验码

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§9.1 软件打包要求 f）g）** |
| **原文** | "f）**安装包必须使用 GPG 或 SM2 签名，并提供公钥**；g）**安装包必须提供 SHA-256 校验码**" |
| **现状** | Docker 镜像 / JAR 包通常无 GPG/SM2 签名流程 |
| **建议** | CI/CD 流水线增加 GPG 签名步骤；制品库提供 SHA-256 摘要文件 |

### C9 — 兼容性要求（动态库 + 国产化）

| 项目 | 内容 |
|------|------|
| **出处** | 开发规范 **§13 兼容性要求**（第1段、第3段） |
| **原文（动态库）** | "除基础C库以外，**应用不应依赖操作系统提供的任何动态链接库**" |
| **原文（国产化）** | "应用软件应充分考虑国产CPU、操作系统、数据库、中间件、浏览器等信创件环境的兼容性要求，**避免对单一技术路线、特定操作系统发行版、专有接口或特定动态库形成强依赖**" |
| **修订意见参考** | 文件C 第 8 条："'除基础C库以外，应用不应依赖操作系统提供的任何动态链接库'要求较严格，实际开发中可能难以完全满足" |
| **现状** | MySQL 5.7（非国产数据库）；未验证麒麟/统信 OS、达梦/人大金仓数据库适配 |
| **建议** | ① 动态库条款主要针对 C/C++ 应用，Java Fat JAR 自带依赖，引用修订意见争取豁免；② 制定国产化适配计划（优先达梦 DM8，兼容麒麟 V10） |

### C10 — 适配测试评分扣分项

| 测试维度 | 出处 | 指标要求 | 我司风险 |
|----------|------|----------|----------|
| 依赖组件数量 | 测试规范 **§7.2.2 服务端适配评价 第8项 可移植性** | "依赖性：依赖组件数量 **≤5 个**得满分；每增加 1 个扣 1 分" | Spring 生态 jar 依赖远超 5 个，**必然扣分** |
| 安装步骤 | 测试规范 **§7.2.2 可移植性** | "安装复杂度：安装步骤 **≤5 步**得满分" | Docker Compose 一键部署可达标 ✅ |
| 关键响应时间 | 测试规范 **§7.2.1/7.2.2 性能效率** | 关键操作响应时间 **≤2 秒** | 需性能基线测试，高峰时段需验证 |
| 高危漏洞 | 测试规范 **§7.2.2 安全性** 及 **§8 测试流程 第6阶段** | "高危漏洞数量 = 0" | 需做 SCA 扫描（如 OWASP Dependency-Check） |
| MTBF | 测试规范 **§8 测试流程 第7阶段 可靠性与稳定性测试** | "MTBF ≥ 1000 小时" | 需 48 小时以上稳定性压测数据支撑 |
| 功能实现率 | 测试规范 **§7.2.1/7.2.2 功能性** | "功能实现率 ≥ 95% 得满分" | 需提供完整测试用例覆盖率报告 |

> ⚠️ **特别关注"依赖组件≤5"扣分项**：此条出自测试规范 §7.2.2 可移植性评价，建议与测试机构沟通评分口径——明确是"直接运行时依赖服务"（如 MySQL/Redis/Nginx，约 3~5 个）还是"所有 jar 包依赖"（不现实）。可同步推动修订意见明确定义口径。

## 六、优先行动建议

1. **立即（0成本）**：在修订意见征集期，就 **C1（Java版本）、C2（Q前缀）、C10（依赖数量口径）** 提交书面反馈，影响标准最终版本。

2. **近期（1-3个月）**：修复 **C5（密码明文）、C7（日志配置）**，成本低，安全评审中同样必要。

3. **中期（3-6个月）**：**C4（HTTPS）、C3（i18n）、C8（包签名）** 纳入下个版本迭代。

4. **长期（视项目需要）**：**C1（JDK升级）、C9（国产化适配）** 结合具体电力项目需求推进。
